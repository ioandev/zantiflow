# ADR-0046 — CI drift guard for the committed LICENSES manifests (reproducible scopes only)

- **Status:** Accepted (implemented) — **amended by [ADR-0047](0047-license-drift-guard-warns-not-fails.md)**
  (the drift step now **warns** instead of failing the build; scope/reproducibility unchanged)
- **Resolves:** [ADR-0036](0036-apache-2-0-and-third-party-license-compliance.md) Open Question/Risk 2
  ("No drift check in CI"); also completes its Risk 1 (stray `MIT` license fields) and Risk 3
  (half-applied `ioandev → ioandev` account rename) as mechanical follow-ups
- **Builds on:** [ADR-0036](0036-apache-2-0-and-third-party-license-compliance.md) (the Apache-2.0
  decision + `scripts/generate-licenses.mjs` and the committed manifests),
  [ADR-0021](0021-dockerization-and-deployment.md) (images bundle the notices — a stale manifest ships
  stale attribution)
- **Date:** 2026-07-12
- **Deciders:** project owner
- **Tags:** ci, licensing, compliance, tooling

## Context

ADR-0036 committed three generated third-party attribution manifests (`LICENSES`, `apps/web/LICENSES`,
`apps/plugin/LICENSES`), regenerated by `just license`, and named as its own Risk 2 that **nothing
stops them drifting**: a contributor who bumps a dependency but forgets to rerun `just license` leaves
the committed manifest stale, and — because those files ship inside every Docker image (ADR-0021) and
on the plugin GitHub Release (ADR-0022) — a distributed artifact then carries stale attribution. The
plugin-release workflow already regenerates the plugin manifest *fresh at release time*, so the
shipped `.wasm` can't go stale; the gap is the **committed** copies, checked by neither `ci.yml` nor
`tests.yml`.

The obvious guard — `just license && git diff --exit-code` — is **not reliably reproducible for the
whole repo** in CI. `scripts/generate-licenses.mjs --scope all` collects three ecosystems and
**silently skips any that is absent** (degrades to a warning), and its PyPI section is read from the
two bots' `.venv`s, whose `apps/{discord,telegram}-bot/requirements.txt` use **unpinned `>=` ranges**
(`discord.py>=2.7`, `aiogram>=3.29`, …). A CI run months later installs whatever those ranges then
resolve to, so the regenerated root manifest legitimately differs from the committed one — a **false
positive** unrelated to any contributor's mistake. The other two scopes have no such problem: `web` is
pure npm resolved from the **frozen `pnpm-lock.yaml`** (pnpm pinned via `packageManager`), and
`plugin` is the Cargo closure resolved from the **committed `Cargo.lock`**; both regenerate
**byte-identically** (verified locally: `web` = 114 npm packages, `plugin` = 135 crates, `diff` empty).

## Decision Drivers

- **No false positives.** A guard that fails on upstream releases nobody touched trains contributors
  to ignore it — worse than no guard.
- **Cover the common drift vector.** The manifests most often go stale when npm (web) or Cargo
  (plugin) deps are bumped; both are lockfile-reproducible.
- **Reuse existing tooling** (`scripts/generate-licenses.mjs` + the toolchains CI already sets up) — no
  new scanner, and no repinning of the bots as a precondition.
- **Fail-closed where reliable, honest where not.** Guard what regenerates deterministically; do not
  pretend to guard what currently can't, and record why plus the prerequisite to fix it.

## Considered Options

1. **Guard only the byte-reproducible scopes (`web` + `plugin`)** *(chosen)* — a `license-drift` CI job
   regenerates both from their committed lockfiles and `git diff --exit-code`s them. Reliable today,
   zero false positives, covers the npm-web and Cargo-plugin drift vectors. Leaves the root manifest's
   npm/Cargo halves unguarded (a backend-only npm bump is not caught — accepted as a known gap).
2. **Guard all three scopes, venvs and all** — rejected: the unpinned bot `requirements.txt` make the
   root PyPI section non-reproducible, so this false-positives on any upstream bot-dependency release;
   also the slowest (recreates two venvs each run).
3. **Pin the bot requirements first, then guard all three** — deferred: pinning the bots (exact `==` /
   a lockfile) is a separate dependency-management decision with its own trade-offs and its own ADR; it
   is the *prerequisite* to a full-root guard, not part of this one.
4. **No guard (status quo)** — rejected: ADR-0036 Risk 2 stands and stale attribution can ship.

## Decision

Add a **`license-drift`** job to `.github/workflows/tests.yml` (the reusable suite that `ci.yml` runs
on PRs/feature pushes **and** `docker-publish.yml` runs as its pre-publish gate — so the guard runs in
both places from one definition). It:

1. sets up Node + pnpm, runs `pnpm install --frozen-lockfile`, and regenerates `apps/web/LICENSES` via
   `--scope web`;
2. sets up the Rust `wasm32-wasip1` toolchain and regenerates `apps/plugin/LICENSES` via
   `--scope plugin` — gated on `needs.detect.outputs.rust == 'true'` (the repo's existing "only run
   stacks that exist" pattern), so it is forward-safe if the plugin is ever absent;
3. `git diff --exit-code`s those two paths, failing with an actionable `::error::` that tells the
   contributor to run `just license` and commit the result.

The **root `LICENSES` is intentionally not diffed** by this job, because its PyPI section is not
reproducible while the bots' deps are unpinned (see Context). This carve-out is documented in the
job's own comment so it is not mistaken for an oversight.

Two mechanical follow-ups ADR-0036 flagged in the same breath are also completed in this task (they
*execute already-recorded decisions*, they are not new decisions): **Risk 1** —
`apps/backend/package.json` and `apps/plugin-dist/package.json` `"license"` fields changed
`MIT → Apache-2.0`, matching the repo-wide Apache-2.0 decision; **Risk 3** — the
`ioandev → ioandev` account rename is finished across the three `@zantiflow/*`
package.jsons, `apps/web/lib/links.ts`, `apps/plugin-dist` (README / config default / `.env.example`),
and the docs pages (`plugin-getting-started.md`, `troubleshooting.mdx`, `astro.config.mjs`,
`index.mdx`, `adrs.mdx`). The ADRs that reference the old handle are left unchanged (immutable
historical record). Separately, `SELF_HOST_SECRET` was added to `apps/backend/.env.example`, closing
the ADR-0035 deploy-discoverability note (`deploy/.env.example` already documented it).

## Consequences

**Positive**
- A forgotten `just license` on an npm-web or Cargo-plugin dependency bump now **fails CI** before
  merge and before any image/Release ships a stale manifest.
- Reliable: the guarded scopes regenerate byte-identically from committed lockfiles → no spurious
  failures.
- No new tooling and no precondition repinning; reuses the generator and CI toolchains already present.

**Negative / costs**
- The **root** manifest's npm + Cargo halves remain unguarded (a backend-only npm bump, or any
  non-web/non-plugin dep, can still leave root stale until the next manual `just license`).
- Adds a CI job (Node install + a Rust toolchain + a `cargo metadata` fetch) to every PR.

**Neutral**
- Full-root coverage stays open pending the bot-dependency pinning decision (see Risks).

## Open Questions / Risks

1. **Pin the bot requirements to enable a full-root guard.** Once `apps/{discord,telegram}-bot` deps
   are pinned (exact `==` / a lockfile), the root `--scope all` manifest becomes reproducible and this
   job can add it (set up both venvs, regenerate, diff). That pinning is its own decision/ADR.
2. **Toolchain-version reproducibility.** The guard assumes the CI pnpm (via `packageManager`) and a
   stable Cargo produce the same manifest bytes as a committer's `just license`. A pnpm major bump that
   changed `pnpm licenses --json` output, or a Cargo change to crate-source layout, could cause a
   one-off diff — resolved by regenerating and committing, the same remedy the guard already prints.
