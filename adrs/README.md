# Architecture Decision Records

This directory holds **Architecture Decision Records (ADRs)** for the zantiflow monorepo. Each ADR
captures one significant decision — its context, the options considered, the choice, and its
consequences — so the reasoning survives beyond the moment it was made.

## Conventions

- **Filename:** `NNNN-kebab-case-title.md`, zero-padded sequential number starting at `0001`
  (e.g. `0001-zellij-session-telemetry-architecture.md`).
- **Immutable-ish:** once Accepted, an ADR is not rewritten. To change a decision, add a new ADR and
  set the old one's status to `Superseded by ADR-NNNN`.
- **Statuses:** `Proposed` | `Accepted` | `Declined` | `Superseded` | `Deprecated`.

## Template (MADR-lite)

```markdown
# ADR-NNNN — <title>

- **Status:** Proposed | Accepted | Superseded | Deprecated
- **Date:** YYYY-MM-DD
- **Deciders:** <who>
- **Tags:** <comma-separated>

## Context
<the problem and forces at play>

## Decision Drivers
<the criteria that matter>

## Considered Options
<options, each with a one-line trade-off; mark the chosen one>

## Decision
<what we chose and how it works>

## Consequences
<positive / negative / neutral outcomes>

## Open Questions / Risks
<what is unresolved or risky>

## References
<links to sources>
```

- [ADR-0006 — Notifications: web-push (PWA) for all, Discord + Telegram for pro](0006-notifications-web-push-and-channels.md) — *Accepted* — delivers ADR-0005 triggers; free = browser Web Push via an installable PWA (custom pre-permission popup + button; install incentivized, required on iOS); pro = Discord + Telegram bot DMs (linked via a website `/link` token, bot↔backend over WebSocket — see ADR-0007). Self-hosters run their own bots.
- [ADR-0008 — Status website: the account dashboard, read API, and live updates](0008-status-website-dashboard.md) — *Accepted* — extends the ADR-0006 PWA into a Next.js dashboard showing machines → sessions → tabs → panes live (privacy-honored, attention badges); defines the tenant-scoped read API + an SSE live stream; **retention = latest state only (no history)**, resolving ADR-0003's open question.
- [ADR-0009 — Durable notification delivery: MariaDB-backed queue, per-channel ack, replay & cleanup](0009-durable-notification-delivery.md) — *Accepted* — persists each delivery as a **MariaDB** row (one per channel), **acked** on success, **replayed** on bot/backend restart (idempotent via `deliveryId`), pruned by a **cron** (default 6h, configurable); DB-as-queue via `SKIP LOCKED`. Picks MariaDB as the backend datastore (resolves ADR-0003); amends ADR-0006/0007.

Amends ADR-0007.
- [ADR-0011 — Tiers & monetization](0011-tiers-and-monetization.md) — *Accepted* — backs the `tier` hook: **GitHub Sponsors** donations (support only, no PRO) + **promo codes** that grant `tier=pro` for a duration (`PromoCode`/`PromoRedemption`, rate-limited). - [ADR-0025 — Re-adopt `claude.thinking` via the status-spinner line](0025-claude-thinking-attention.md) — *Accepted* (partly supersedes ADR-0005) — revives the `claude.thinking` attention ADR-0005 had dropped, detecting Claude Code's working state in a `claude` pane's visible tail (~15 lines): a `Gerund…` spinner **plus** a status anchor (`esc to interrupt` / `still thinking with <effort> effort`). The two sit on **different lines** in the default TUI (gerund on the spinner line, `esc to interrupt` on the footer bar), so the match is *cross-line*, not same-line — precise (prose gerunds don't trigger it) yet actually matching the real layout, not the brittle TUI parsing ADR-0005 rejected. An explicit prompt still wins (needs-input precedence); local-only scan, only the `type` leaves; wire contract unchanged (v4). The dashboard shows a **distinct "thinking" indicator**, separate from (and excluded from) the "needs attention" count.
- [ADR-0024 — Opt-in real-hostname lookup](0024-opt-in-hostname-lookup.md) — *Accepted* (amends ADR-0002) — the real hostname (and the `RunCommands` permission it needs) becomes **opt-in, OFF by default** via a new plugin `hostname` flag; gated by `wants_hostname()` (`hostname=on` AND `machine=real`). `RunCommands` is requested **lazily** in a two-phase handshake so alias/hidden/default users are never prompted for it, and a `RunCommands` denial never disables core telemetry. Resolves the ADR-0002 least-privilege audit finding.
- [ADR-0026 — Minimise plugin→backend updates (change-driven, presence-aware sends)](0026-minimise-plugin-update-cadence.md) — *Proposed* (amends ADR-0001/0005/0008/0016; amended by ADR-0029) — the plugin stops POSTing every second: it sends **only on change** (coalesced ~30 s idle / ~15 s watched), keeps a ~30 s **keepalive** while an attention is active so notifications still fire within the ~2 min budget, and turns the 5 s output poll into an **always-on control channel** carrying a per-session **liveness touch** (so quiet-but-live sessions don't drop under the existing 60 s read-filter) + **viewer-presence** (SSE connection primary, explicit 15 s view-request fallback, rate-limited refresh button). Firing stays ingest-triggered (no new backend sweep); ingest wire **unchanged (v4)**.
- [ADR-0027 — Machine-level `claude.idle`: notify when all Claude sessions go quiet](0027-machine-idle-claude-attention.md) — *Accepted* (partly supersedes ADR-0005; amends ADR-0026) — a new **backend-derived, machine-scoped** attention that fires when **every Claude pane on a machine** has produced no output for longer than the tier threshold (**1 min pro / 5 min free**). It is the first attention computed server-side (implements ADR-0005 §8) because "all sessions idle" is cross-session and no single per-session plugin instance can see it; and because ADR-0026 stops an idle machine's ingests, it fires from a **~20 s backend sweep** over liveness-touched, non-stale state (not at ingest). Reuses the episode engine (self-timed, `threshold 0` + 300 s cooldown + clear-on-resume) and the existing notification/SSE plumbing; text is "All Claude sessions are idle". Wire contract **unchanged (v4)**.
- [ADR-0028 — Drop `session.detached`; add machine-level `machine.offline`](0028-machine-offline-attention.md) — *Accepted* (partly supersedes ADR-0005; amends ADR-0027) — retires the noisy `session.detached` notification (a session with no attached client is normal usage, not an alert — the backend now filters it out of ingest) and replaces it with the signal owners actually want: **`machine.offline`**, a backend-derived, machine-scoped attention that fires **"A machine went offline"** when a machine stops reporting past `STALE_AFTER_MS` (60 s; genuine because ADR-0026's control-poll keeps a live machine fresh). Computed by the shared ~20 s `sweepMachineAttentions` (both machine-level attentions emitted in one call), self-timed with a long cooldown so it fires **once per disconnect**, clears on reconnect, and is bounded to a 15-min lookback. Excluded from the "needs attention" count. Wire **unchanged (v4)**.
- [ADR-0029 — Opt-in long-poll for the control channel](0029-optin-longpoll-control-channel.md) — *Accepted* (amends ADR-0026, ADR-0016; default flipped by ADR-0031) — an `control_long_poll` plugin flag that asks the backend to **hold** each control poll open (additive `waitMs` on the control request) until a pane-output request or manual refresh wakes it, dropping that latency from up to ~5 s to ≈1 s while sending *fewer* requests. A true server push is impossible from a plugin (`web_request` is single-buffered, no socket — FINDINGS §5), so this is HTTP long-polling with a plugin-side **watchdog** that re-issues a dropped hold; the fixed ~5 s poll stays the default and fallback. Depends on the Zellij host actually holding a long request (unverified) → gated on the real-Zellij smoke before it can be enabled. Ingest wire **unchanged (v4)**.
- [ADR-0030 — Pane output is ephemeral (fresh-on-open, not persisted)](0030-ephemeral-pane-output.md) — *Accepted* (amends ADR-0016, ADR-0018; **storage superseded by ADR-0032**) — fixes the backend serving a **stale cached** pane capture: `readOutput` now gates on the request lifecycle (returns `pending` until a **new** capture for the current request lands, never a previous one), `registerRequest` **drops the prior snapshot**, and a prune sweep makes stored output **ephemeral (~2 min)** so scrubbed content doesn't linger between views. The dashboard drawer is a one-shot fetch holding its lines in component state. Wire **unchanged (v4)**.
- [ADR-0031 — Long-poll control channel is ON by default](0031-longpoll-control-default-on.md) — *Accepted* (amends ADR-0029) — flips the plugin `control_long_poll` default **OFF → ON** so pane-output/refresh latency is ≈1 s (not up to ~5 s) out of the box, and tightens the dashboard drawer's pending poll 2 s → 1 s. Safe because long-poll **self-degrades**: if the host won't hold the request it becomes a ~1 s poll (still faster than the 5 s fixed poll); only a host that *silently drops* held requests regresses (→ ~35 s watchdog, visible on reload), and `control_long_poll off` reverts to the fixed poll. Takes effect after a plugin `.wasm` rebuild + reload. Wire **unchanged (v4)**.
- [ADR-0032 — Pane output is never persisted (in-memory relay, not the DB)](0032-pane-output-never-persisted.md) — *Accepted* (partly supersedes ADR-0030; amends ADR-0016, ADR-0018) — moves captured pane content from an ephemeral MariaDB row to a **process-local in-memory `PaneOutputStore`** that is **never written to the database at all**, so the "never stored" / "content otherwise never leaves your machine" promise is literally true. The **`PaneOutput` table is dropped**; the request lifecycle (`OutputRequest` — which pane, pending/fulfilled, no content) stays in the DB, and the whole ADR-0030 lifecycle (fresh-on-open, drop-on-re-request, ~2 min prune, purge-on-forget) is unchanged, now against memory. Joins presence/waiters/auto-refresh as in-process single-backend state (ADR-0019). Wire **unchanged (v4)**.
- [ADR-0033 — Spotlight: a PRO-only live album of active Claude sessions](0033-spotlight-active-claude-album.md) — *Accepted* (amends ADR-0016) — a new PRO-only `/spotlight` page + hard PRO-gated `GET /api/v1/spotlight` (the first route PRO fully gates, returning **`403 requires_pro`**) showing **every active Claude session across all the account's machines** as an album of "photos" you flip through, each streaming its pane output (reusing the ADR-0016 pane-output loop, extracted to a shared `usePaneOutputStream` hook; only the on-screen photo streams). **"Active" = running (not exited)**; the backend returns only the currently-active roster and the client tracks **completed** sessions by diffing successive rosters (kept, greyed, until "Clear sessions that completed"); new sessions appear on their own via the SSE stream. Reliable Claude detection (pane-NAME marker) is promoted to `@zantiflow/protocol` for the backend roster (the web keeps a test-pinned copy — it carries no `@zantiflow/*` dep). Wire **unchanged (v4)**.
- [ADR-0034 — Reliable `claude.thinking`: pane-title marker + freshness, and a dashboard marker strip](0034-reliable-claude-thinking-marker-freshness.md) — *Accepted* (partly supersedes ADR-0025) — fixes a Claude pane showing a **stuck "`.`"** on the dashboard after a turn ends. Two bugs: the `claude.*` detectors gated on the pane **`command`** (Zellij reports it `null`) and read the **tab name** for the spinner (the marker is on the **pane title**), so they **never fired**; and Claude Code leaves the last Braille spinner frame (`⠂`, renders as a dot) **frozen** in a background pane's title, which the change-driven cadence (ADR-0026) then has no reason to re-send. Fix: detect Claude by the **pane-title marker** (command as fallback); emit `claude.thinking` only when a spinner-marked pane is **still producing output** (a per-pane content-freshness tracker, `THINKING_STALE_TICKS`), so a frozen glyph **self-clears** (a structural change the send-gate pushes) instead of sticking; **strip the marker from the dashboard/Spotlight display** (`paneDisplayName`); and gate the Spotlight `thinking` flag on the same freshness. Best-effort, fails closed under redaction. Takes effect after a plugin `.wasm` rebuild + reload. Wire **unchanged (v4)**.
- [ADR-0035 — Self-host owner sign-in via a configured secret](0035-self-host-owner-sign-in-secret.md) — *Accepted (implemented)* (amends ADR-0004) — **written retroactively** (the backend cited "ADR-0035" in ~12 places before this file existed): an optional, self-host-only owner-auth path — set `SELF_HOST_SECRET` and log in with that single value instead of registering a Google OAuth app. Adds `GET /auth/methods` (one web image renders the right login surface) + `POST /auth/local` (mounted only when the secret is set; own strict rate-limit bucket; wrong secret → `401 invalid_secret`; success sets the same `ztf_session` cookie, returns `204`). The secret is verified in-memory, timing-safe, **never stored**, and must differ from `TOKEN_SECRET`; the local owner is a normal `Account` under the reserved `('local','owner')` identity. Owner plane only; ingest wire unchanged (v4).
- [ADR-0036 — Apache-2.0 licensing & automated third-party license-compliance manifests](0036-apache-2-0-and-third-party-license-compliance.md) — *Accepted (implemented)* (corrects ADR-0022 §1's "MIT/OSS"; relates to ADR-0021; Risks 1–3 resolved by ADR-0046) — **written retroactively**: licenses the whole polyglot repo under **Apache-2.0** (`LICENSE` + `NOTICE`; declared across the root `package.json`, `apps/plugin/Cargo.toml`, and every `@zantiflow/*` package) and adds a deterministic third-party attribution system — `scripts/generate-licenses.mjs` (`--scope all|web|plugin`) drives `pnpm licenses` + `cargo metadata` (wasm closure) + each bot venv, emits a **fail-closed compliance overview** flagging any GPL/AGPL/unrecognised license, and writes committed `LICENSES` manifests (regenerated via `just license`) that ship inside every Docker image and on the plugin GitHub Release.
- [ADR-0037 — Host-shared plugin identity via `/cache`](0037-host-shared-plugin-identity-via-cache.md) — *Accepted (implemented)* (amends ADR-0003, ADR-0012, ADR-0022; refines ADR-0002) — **written retroactively**: records that Zellij runs **one plugin instance per session**, so identity (fingerprint salt, `machineId`, paired ingest token) is persisted in host-shared **`/cache`** — not the per-session `/data` those ADRs specified — so the whole host reports as **one machine** and you **pair once**. `machineId` is seeded from the **salted hostname** (stable across cache-clears/reinstalls, random fallback), closing ADR-0003's "wipe → history splits" gap; `sid`s are deterministic across instances via the shared salt; pane-output requests are served only by the owning instance.
